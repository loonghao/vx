# 2026-03-19

## Fixed yq provider.star — duplicate assignment bug

The yq `provider.star` had a critical bug: after defining custom `download_url` and `install_layout` functions (lines 77-94), lines 96-102 re-assigned ALL variables from the template `_p`, completely overriding the custom functions. This caused:

1. **Lint warnings**: "Unused assignment" for the first batch of template assignments (lines 70-74) since they were immediately overridden by lines 96-102.
2. **CI failure**: The template's default `install_layout` placed the binary at `store/yq/4.52.4/windows-x64/yq.exe` but `get_execute_path` expected it at `store/yq/4.52.4/bin/yq.exe`, causing "Executable not found" error.

Fix: Removed the duplicate assignment block. Now template functions are assigned once, then `download_url` and `install_layout` are defined as custom overrides (their def names become the final exports). Also added `source_name` to `install_layout` to properly rename `yq_windows_amd64.exe` → `yq.exe`, following the same pattern as `hadolint` and `rcedit` providers.

## Fixed xmake provider.star — download URL 404 on Windows

The xmake `download_url` generated `xmake-bundle-v3.0.7.win64exe` (missing `.` before `exe`). Root cause: the `_XMAKE_PLATFORMS` dict had `ext="exe"` without a leading dot for Windows. Fixed by changing to `.exe`. Also fixed `install_layout` to use `__type` instead of `type`.

## Fixed wix CI test failure — system-installed tool not found

The wix provider installs via winget (system install), which returns `executable_path = "system"`. The test handler then tried to construct a file path using the version string `"system (winget)"` as a directory component, producing a bogus path like `~/.vx/store/wix/system (winget)/windows-x64/wix.exe`.

Three fixes:
1. **Test handler** (`vx-cli/src/commands/test/handler.rs`): When executable not found and version contains "system", try `find_runtime_executable_for_test` as a fallback (same pattern already used for failed installs).
2. **wix provider.star**: Added `system_paths` for WiX v4 (`C:/Program Files/WiX Toolset v4*/bin/wix.exe`) so the runtime system can locate the executable after winget install.
3. **Starlark execute.rs**: Added `"binary"` as alias for `"binary_install"` in the `install_layout` type matcher, consistent with `"archive"` being an alias for `"archive_install"`. When `"binary"` layout has no URL, maps to `Archive` layout hint.

Also fixed vcpkg's `install_layout` to use `__type` instead of `type` (same warning).

## Fixed Linux CI failures — round 2

Commit `69d0bf42`: Fixed 6 more CI failures on Linux.

### ffplay/ffprobe (ffmpeg provider)
**Root cause**: `bundled_runtime_def` without explicit `test_commands` defaults to `{executable} --version`. FFmpeg tools use `-version` (single dash), not `--version`. Additionally, the johnvansickle.com static builds are compiled with `--disable-ffplay`, so ffplay may not exist.
**Fix**: Added explicit `test_commands` with `-version` flag and matching `expected_output` patterns for both bundled runtimes.

### gofmt (go provider)
**Root cause**: gofmt has no `--version` flag. The default test `gofmt --version` fails with non-zero exit code.
**Fix**: Set `test_commands = []` to skip functional tests. With empty test_commands, the Starlark runtime returns `None` for test config, and `run_default_test()` kicks in — which passes if ANY output exists (even on stderr).

### lld (llvm provider)
**Root cause**: `lld` is a multi-call binary that expects a `-flavor` argument. Plain `lld --version` fails.
**Fix**: Set `test_commands = []` — same approach as gofmt.

### xmake on Linux
**Root cause**: `install_layout` returned `archive` type for Linux, but the downloaded asset `xmake-bundle-v3.0.7.linux.x86_64` is a self-extracting binary, not a tar.gz. Archive extraction fails → executable not found.
**Fix**: Changed Linux install_layout to `binary` type (same as Windows).

### yq on Linux
**Root cause**: `install_layout` returned a dict without `__type` field. The execute.rs type matcher got empty string `""`, hit the `other` branch → "Unknown install_layout descriptor type, ignoring" → returned None → install hints lost.
**Fix**: Added `"__type": "binary_install"` to the install_layout return dict.

### imagemagick lint warnings
**Fix**: Removed unused `github_asset_url` import, prefixed unused `ctx`/`version` params with `_`.

## Fixed CI failures — round 4: system_install closure detection + rez

Commit `27a3127b`: Fixed system_install closures not being parsed and rez using unrecognized managers.

### Root cause: `multi_platform_install`/`cross_platform_install` closures not detected
When `system_install = multi_platform_install(...)` is used at module level, it creates a Starlark closure. `parse_system_install_strategies` calls `get_variable("system_install")` which serializes the closure to a JSON string `"<function _system_install>"`. The code then tries `starts_with("<function")` → calls `call_function("system_install", ctx)`. However, this path may fail or return empty strategies in some cases (lint is run twice — once per get_variable and call_function — causing duplicate warnings).

**Fix**: Converted ALL module-level closure assignments to explicit `def system_install(ctx):` functions:
- **imagemagick**: `multi_platform_install(...)` → `def system_install(ctx):`
- **ffmpeg**: `multi_platform_install(...)` → `def system_install(ctx):`
- **bash**: `multi_platform_install(...)` → `def system_install(ctx):`
- **make**: `multi_platform_install(...)` → `def system_install(ctx):`
- **nasm**: `cross_platform_install(...)` → `def system_install(ctx):`

### rez: "No installation strategy available" on all platforms
**Root cause**: `system_install` used `"manager": "uvx"` and `"manager": "pip"` which aren't registered in `PackageManagerRegistry`. Only standard system package managers (brew, apt, winget, etc.) are recognized.
**Fix**: Converted rez to RFC 0033 `package_alias` pattern (like meson/conan/pre-commit):
- Added `package_alias = {"ecosystem": "uvx", "package": "rez"}`
- Switched from `github_permissions` to `system_permissions(extra_hosts=["pypi.org"], exec_cmds=["uvx", "uv"])`
- Changed dep from `python >= 3.8` to `uv` (since rez runs via uvx)
- Removed broken `system_install` and `fetch_versions`

### Key learning: provider.star `system_install` patterns (UPDATED)
- ✅ `system_install = system_install_strategies([...])` — static dict, BEST approach. Platform filtering handled by per-manager helpers' `platforms` field.
- ❌ `def system_install(ctx):` — function, `parse_system_install_strategies` may fail to detect/call it reliably
- ❌ `system_install = multi_platform_install(...)` — closure, same issue
- ❌ `system_install = cross_platform_install(...)` — closure, same issue
- For Python tools: use `package_alias` (RFC 0033) not system_install with pip/uvx

## Fixed macOS CI failures — round 5: system_install static dict + binary_install metadata loss

### Problem 1: ffmpeg/ffprobe/ffplay "No installation strategy available" on macOS
**Root cause**: The `def system_install(ctx):` function pattern (set in round 4) was unreliable. `parse_system_install_strategies` in `builder.rs` has two code paths: (1) static dict with `"strategies"` key → direct parse (works), (2) function detected via `"<function"` prefix → `call_function` (unreliable on some platforms).
**Fix**: Converted ALL remaining `def system_install(ctx):` functions to static dict pattern `system_install = system_install_strategies([...])`. Affected providers: ffmpeg, imagemagick, bash, make, nasm. The per-manager helpers (`brew_install`, `apt_install`, `winget_install`, etc.) automatically set a `platforms` field for filtering.

### Problem 2: xmake "Executable not found" at expected path on macOS
**Root cause 1**: `system_install` used function pattern (same as Problem 1).
**Root cause 2**: `install_layout` returned `{"__type": "binary", ...}` without URL. In `execute.rs`, `binary_install`/`binary` without URL was converted to `InstallLayout::Archive { url: None, ... }`, losing `source_name`, `target_name`, `target_dir` fields. `build_layout_meta()` then returned empty metadata, so the downloaded file was never renamed from `xmake-bundle-v3.0.7.macos.arm64` to `xmake`.
**Fix**: (a) Rewrote xmake `provider.star` completely with proper `binary_install` layout including `source_name`, `target_name`, `target_dir` fields, and static dict `system_install`. (b) Fixed `execute.rs` to return `Ok(None)` for `binary_install` without URL, causing bridge.rs to fall through to `install_layout_raw()` which preserves all dict fields.

### Problem 3: yq "Install: ✓ | Functional: ✗" on macOS
**Root cause**: Same `binary_install` metadata loss issue as xmake. yq's `install_layout` returns `binary_install` without URL, with `source_name=yq_darwin_amd64`, `target_name=yq`, `target_dir=bin`. These fields were lost during the Archive conversion.
**Fix**: The execute.rs change (return `None` for `binary_install` without URL) fixes yq automatically — the raw dict preserves all fields for `build_layout_meta()`.

## Fixed Linux CI failures — round 6: binary_install metadata loss + pnpm + ffplay + pwsh

### Problem 1: pnpm "Executable not found: .../linux-x64/pnpm"
**Root cause**: Three compounding bugs:
1. `install_layout` returned `{"type": "binary", ...}` without `source_name` or `__type`
2. `execute_install_layout_raw` didn't check `target_name`/`target_dir` in `has_useful_fields` → dict dropped
3. Without layout hints, exe path fell back to `install_dir/pnpm` instead of `install_dir/bin/pnpm`
**Fix**: Rewrote pnpm `install_layout` with `__type: "binary_install"`, computed `source_name` from platform suffix (e.g., `pnpm-linux-x64` → rename to `pnpm`).

### Problem 2: execute.rs `execute_install_layout_raw` dead end for binary_install
**Root cause**: When typed path returned `None` for `binary_install` without URL, the raw path saw `__type` field and also returned `None` — creating a dead end where layout metadata was completely lost.
**Fix**: In `execute_install_layout_raw`, when `__type` is `"binary_install"` or `"binary"`, strip the type field and return the raw dict (preserving `source_name`, `target_name`, `target_dir`). Also added `target_name` and `target_dir` to `has_useful_fields` check.

### Problem 3: ffplay "Install: ✓ | Functional: ✗" on Linux
**Root cause**: johnvansickle.com static FFmpeg builds compiled with `--disable-ffplay` (SDL can't be statically linked). Provider incorrectly listed `ffplay` in Linux `executable_paths`.
**Fix**: Removed `ffplay` from Linux `executable_paths` in ffmpeg provider.star.

### Problem 4: pwsh failure on Linux
**Root cause**: `system_install` only had Windows strategies (winget, choco). No fallback on Linux/macOS.
**Fix**: Added `brew_install("powershell")` and `apt_install("powershell")` to pwsh system_install.

### Problem 5: xmake/yq on Linux — same binary_install metadata loss as macOS
**Fix**: Resolved by the `execute_install_layout_raw` fix. No provider.star changes needed.

### Problem 6: magick/convert on Linux
**Fix**: Already fixed locally in round 5 (static dict pattern). Just needs commit + push.

### Problem 7: rcedit on Linux — expected behavior
rcedit is Windows-only (`platforms = {"os": ["windows"]}`). Not a bug — CI configuration issue.
