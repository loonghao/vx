## 新增 cross provider + CI 修改

### 问题
1. CI 中 `vx just cross-build` 失败因为 `cross` 工具没有 vx provider
2. cross-build 和 coverage job 限制只在 main 分支运行，PR 时无法测试

### 修改

#### 新增 `cross` provider (`crates/vx-providers/cross/`)
- cross-rs/cross: Rust 交叉编译工具
- Asset 格式特殊：`cross-{triple}.tar.gz`（不含版本号）
- 仅 x86_64 预编译二进制（macOS arm64 通过 Rosetta 2 使用 x86_64 版）
- Tag 格式: `v{version}`
- 依赖: cargo (Rust 工具链)

#### CI 修改 (`.github/workflows/ci.yml`)
- cross-build: 改为 `is_main != 'true'` 条件，只在 PR 时运行，main 不跑
- coverage: 改为 `is_main != 'true'` 条件，只在 PR 时运行，main 不跑
- cross-build: 移除 `taiki-e/install-action` 安装 cross，改由 vx 自动管理
- ci-success summary: 更新分组标题为 "PR only"

#### actions/upload-artifact 和 download-artifact 升级到 v7
更新了所有 workflow 文件中的 artifact actions：
- ci.yml, benchmark.yml, test-providers.yml, test-docker.yml
- test-action.yml, release.yml, package-managers.yml, maintenance.yml
- docker-publish.yml, linux-packages.yml, examples-msvc.yml
- test-providers.yml.example
总计 52 处引用全部从 v5/v6 更新到 v7

#### 修复 Required Status Checks 卡住问题
问题：`Action Success`、`Docker Success`、`Provider Success` 三个 required checks 在 PR 没有触碰对应路径时永远停在 "Waiting for status to be reported"。
原因：这三个 workflow (test-action.yml, test-docker.yml, test-providers.yml) 的 `pull_request` 触发器使用了 `paths` 过滤器，没触碰路径时 workflow 不运行，success gate job 不报告。
解决方案：
1. 移除 `pull_request` 触发器的 `paths` 过滤器（`push` 保留 paths）
2. 添加 `check-changes` job 使用 `dorny/paths-filter@v3` 检测变更
3. 所有实际工作 job 加上 `if: needs.check-changes.outputs.should_run == 'true'`
4. success gate job 在无变更时直接 `exit 0`（成功），确保 required check 始终有报告

## 修复 openclaw 和 oxlint CI 失败

### openclaw 修复（3 个问题）
1. **lint**: `fetch_versions(ctx)` → `fetch_versions(_ctx)` (未使用参数)
2. **fetch_versions 格式错误**: 返回 `{"type": "npm_registry", ...}` 是无效格式，Rust 端不认识。改为使用 `fetch_json_versions(ctx, url, "npm_registry")`（正确的 `__type: "fetch_json_versions"` 描述符）
3. **permissions**: 添加 `extra_hosts = ["registry.npmjs.org"]` 允许访问 npm registry

### oxlint 修复（executable 路径不匹配）
- **根因**: oxlint 的 tar.gz 内文件名是 `oxlint-{triple}`（如 `oxlint-x86_64-unknown-linux-gnu`），而不是 `oxlint`
- `archive_layout("oxlint")` 生成 `executable_paths: ["oxlint"]` — 找不到
- **修复**: 
  - 自定义 `install_layout` 在 `executable_paths` 中包含 `[dst_name, src_name]`（即 `["oxlint", "oxlint-{triple}"]`）
  - 自定义 `get_execute_path` 返回 triple 命名的实际路径 `{install_dir}/oxlint-{triple}`
  - 移除未使用的 `path_fns` import，替换为 `exe_suffix` import
