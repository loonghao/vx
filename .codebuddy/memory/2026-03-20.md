## fix: resolve .cmd executables for bundled runtimes on Windows

**Problem**: `vx npm --version` failed on Windows with "os error 193 - not a valid Win32 application". The executable search found the Unix shell script `npm` (bare name, no extension) instead of `npm.cmd`.

**Root cause**: Three search functions only tried `.exe` extension on Windows, missing `.cmd`:
1. `find_executable_recursive` in `detection.rs` - recursive dir search
2. `prepare_execution` in `manifest_runtime/mod.rs` - direct candidate list for bundled runtimes
3. `convention_execute_path` in `handle.rs` - convention-based path scanning

**Fix**: All three functions now search for `.exe`, `.cmd`, and bare name in priority order on Windows. The recursive search was rewritten to collect all matching files first, then return the best match by priority (`.exe` > `.cmd` > bare name).

## fix: defense-in-depth for Windows extensionless executable resolution

**Problem**: CI still failing with `vx npm --version` on Windows Server 2025 - OS error 193 when trying to execute bare `npm` (Unix shell script) instead of `npm.cmd`.

**Analysis**: Traced the full execution pipeline (ResolveStage → EnsureStage → PrepareStage → ExecuteStage). All search functions (`find_executable_in_dir`, `find_executable_recursive`, `prepare_execution`) correctly prioritize `.exe` > `.cmd` > bare name on Windows. The exact upstream cause of the bare `npm` reaching execution is hard to pinpoint (possibly ExecPathCache, ResolutionCache, or filesystem timing).

**Fix** (defense-in-depth at two layers):
1. `build_command()` in `command.rs`: Added `resolve_windows_executable()` that checks for `.cmd`/`.exe`/`.bat` variants when an extensionless executable is about to be spawned. If `npm.cmd` exists alongside bare `npm`, it transparently switches to `npm.cmd` and routes through `cmd.exe /c`.
2. `prepare_execution()` in `manifest_runtime/mod.rs`: Added `prefer_windows_executable()` that validates found executables have a Windows-compatible extension before returning them. Applied to both direct candidate search and recursive search results.

## fix(system-tools): add 5s timeout to version detection

**Problem**: `test_list_system_tools` and `test_list_system_tools_all` hanging 2000+ seconds on Windows CI. Root cause: `get_tool_version()` in `system_tools.rs` spawns subprocesses (`--version`, `-V`, `-v`, `version`) with no timeout. On Windows, some tools (Python/Java wrappers, GUI tools) hang indefinitely.

**Fix** (in `crates/vx-cli/src/system_tools.rs`):
1. Added 5-second timeout per subprocess using `spawn()` + `try_wait()` polling
2. Reduced version flags from 4 to 2 (only `--version` and `-V`; removed `-v` which is often verbose mode, and `version` subcommand)
3. Redirect stdin to null to prevent interactive prompts
4. Set `CREATE_NO_WINDOW` on Windows to avoid GUI pop-ups
5. Kill and reap timed-out child processes
